برومبت استعلام التكامل بين تطبيق الموبايل والسيرفر
مقدمة
أنا الوكيل المسؤول عن تطبيق الموبايل (React Native / Expo) لنظام MPBF (مصنع الأكياس البلاستيكية الحديثة). التطبيق يتصل بالسيرفر الخلفي على https://modern-mps.replit.app. قمت بمراجعة شاملة للكود وإصلاح عدة أخطاء. أحتاج منك كمطور السيرفر أن تراجع كل ما يلي وتخبرني بأي نواقص أو اختلافات بين ما يتوقعه التطبيق وما يقدمه السيرفر فعلياً.

1. الإصلاحات التي تمت على تطبيق الموبايل
أ) إصلاح مشكلة رفع الصور بدون توكن (lib/api.ts)
المشكلة: دالة mobileUploadApi.uploadImage كانت ترسل Authorization: Bearer null عندما لا يكون هناك توكن مخزن، مما يسبب فشل 401 صامت.
الإصلاح: أضفت فحص if (!token) قبل إرسال الطلب. إذا لم يوجد توكن، يتم استدعاء onUnauthorized() ورمي خطأ SESSION_EXPIRED.
السؤال للسيرفر: هل endpoint رفع الصور (POST /api/mobile/upload/image) يتعامل بشكل صحيح مع التوكنات المنتهية ويرجع 401؟ وهل يدعم تجديد التوكن بعد رفع صورة فاشل؟
ب) إصلاح non-null assertions في شاشات الإنتاج
المشكلة: في app/production/film.tsx، كان expandedPO! يُستخدم بدون حماية في mutation function لإنشاء رول جديد. نفس المشكلة في app/production/cutting.tsx مع cuttingRollId!.
الإصلاح: أضفت فحوصات null صريحة (if (!expandedPO) throw new Error(...)) قبل استخدام القيم.
السؤال للسيرفر:
POST /api/rolls/create-with-timing - هل يتحقق من أن production_order_id موجود وصالح؟ وماذا يرجع إذا كان غير صالح؟
POST /api/rolls/{id}/complete-cutting - هل يتحقق من أن الرول موجود وفي مرحلة القطع فعلاً؟
ج) إضافة customer_name_ar لنوع ProductionOrder
المشكلة: عدة شاشات تستخدم po.customer_name_ar لعرض اسم العميل بالعربي، لكن الحقل لم يكن موجوداً في تعريف TypeScript.
الإصلاح: أضفت customer_name_ar?: string للـ interface.
السؤال للسيرفر: هل API الـ production orders (GET /api/production-orders) يرجع فعلاً حقل customer_name_ar؟ إذا لا، أرجو إضافته في الاستجابة.

3. ملاحظات مهمة حول شكل البيانات
القيم العددية كنصوص (Strings)
التطبيق يتوقع أن جميع القيم العددية في أوامر الإنتاج ترجع كـ strings:

quantity_kg, overrun_percentage, final_quantity_kg, produced_quantity_kg, printed_quantity_kg, net_quantity_kg, waste_quantity_kg
film_completion_percentage, printing_completion_percentage, cutting_completion_percentage
التطبيق يستخدم parseFloat() لتحويلها عند الحاجة
سؤال: هل السيرفر يرجع هذه القيم كـ strings فعلاً أم كـ numbers؟ إذا كان يرجعها كـ numbers، التطبيق سيعمل بشكل صحيح لكن تعريفات TypeScript ستكون غير دقيقة.

شكل الاستجابة (Response Shape)
التطبيق يتعامل مع شكلين من الاستجابات:

مصفوفة مباشرة: [item1, item2, ...] - مثل /api/customers?all=true
كائن مغلف: { data: [item1, item2, ...], count?, total?, page?, limit? } - مثل معظم الـ endpoints
الدالة normalizeList() تتعامل مع كلا الشكلين تلقائياً.

سؤال: أرجو تأكيد أي endpoints ترجع مصفوفة مباشرة وأيها ترجع الشكل المغلف.

التواريخ
التطبيق يتوقع التواريخ بصيغة ISO 8601 (مثل 2025-01-15T08:30:00Z). هل كل الـ endpoints ترجع التواريخ بهذه الصيغة؟

Rate Limiting
التطبيق يتعامل مع خطأ 429 ويقرأ retry_after_seconds من جسم الاستجابة. هل هذا الحقل موجود في استجابة 429؟

4. أسئلة محددة للسيرفر
حقل customer_name_ar في production orders: هل يُرجع فعلاً من /api/production-orders؟ التطبيق يستخدمه لعرض اسم العميل بالعربي في شاشات الأفلام والطباعة والقطع.

حقل customer_name_ar في enhanced orders: هل /api/orders/enhanced يرجع customer_name_ar؟

Batch creation: هل /api/production-orders/batch يدعم إرسال عدة أوامر إنتاج دفعة واحدة؟ وما هو شكل الاستجابة الناجحة؟

الحضور - حالة اليوم: التطبيق يقارن record.date مع تاريخ اليوم (YYYY-MM-DD). هل السيرفر يرجع date بنفس الصيغة؟

الحضور - الموقع: التطبيق يرسل location: { lat, lng, accuracy, isMocked }. هل السيرفر يتعامل مع كل هذه الحقول؟

Film Queue: هل /api/production/film-queue يرجع FilmQueueItem[] مباشرة أم مغلف بـ { data: [...] }؟

Rolls: هل /api/rolls يرجع مصفوفة مباشرة أم مغلف؟ (التطبيق يستخدم normalizeList للتعامل مع كلاهما)

Machine types: التطبيق يفلتر الماكينات حسب type: 'film' | 'printing' | 'cutting'. هل هذه هي القيم الفعلية؟

أداء المشغلين والماكينات: هل الـ endpoints التالية تعمل وما هو شكل استجابتها؟

/api/production/users-performance/{section}
/api/production/machines-production/{section}
/api/production/rolls-tracking/{section}
Quality Issues: هل إنشاء مشكلة جودة (POST /api/quality-issues) يقبل production_order_id اختياري لربطها بأمر إنتاج؟

Leave Requests - employee_id: التطبيق يرسل employee_id كـ string. هل هذا صحيح أم يجب أن يكون number (user id)؟

التوكن والصلاحيات:

هل permissions[] في كائن المستخدم تتضمن قيمة 'admin' للمسؤولين؟ (التطبيق يستخدم هذا للتحقق من الصلاحيات)
ما هي قائمة الصلاحيات المتاحة التي يمكن أن يحملها المستخدم؟
إنشاء الطلبات: عند إنشاء طلب جديد (POST /api/orders)، التطبيق يرسل order_number من النتيجة المسبقة لـ /api/orders/next-number. هل يمكن أن يحدث تعارض إذا طلب مستخدمان نفس الرقم في نفس الوقت؟

Endpoints مفقودة محتملة: هل هناك endpoints في السيرفر لم يتم استخدامها في التطبيق بعد ويمكن أن تضيف قيمة؟

خطأ 401 مقابل 403: هل السيرفر يفرق بين 401 (غير مصدق) و403 (غير مصرح)؟ التطبيق حالياً يتعامل مع 401 فقط لتجديد التوكن.

5. ملخص البنية التقنية للتطبيق
Framework: React Native + Expo (SDK 52)
Routing: expo-router (file-based)
State Management: React Query (@tanstack/react-query) + React Context
API Layer: fetch-based مع auto token refresh وretry on 401
Storage: AsyncStorage للتوكنات وبيانات المستخدم
اللغة الافتراضية: العربية (RTL)
التاريخ: ar-EG locale (تقويم ميلادي)
Timeout: 20 ثانية لكل طلب API
Proxy (Web): Metro على 8081 + proxy على 5000 يوجه /api/* إلى https://modern-mps.replit.app
المطلوب: أرجو مراجعة كل ما سبق وإخباري بأي:

endpoints غير موجودة أو معطلة
اختلاف في شكل الاستجابة عما يتوقعه التطبيق
حقول مفقودة في الاستجابات (خاصة الحقول العربية _ar)
مشاكل أمنية أو ثغرات في التكامل
endpoints جديدة متاحة لم يتم استخدامها
أي توصيات لتحسين التكامل
