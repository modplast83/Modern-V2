**Task: Build a Mobile App (Expo) as a Separate Project Without Affecting Existing Web App**

You are required to create a **production-ready mobile application using Expo (React Native)** inside the current repository, while ensuring **zero impact on the existing web application and backend codebase**.

### ⚠️ Critical Constraints

* Do NOT modify, refactor, or break any existing web app code.
* Do NOT change the backend structure, APIs, or database schema unless explicitly required and documented.
* The mobile app must be implemented as a **fully separate project** within the same repository.

---

## 1. Project Structure (Mandatory)

Create a clean monorepo-style structure:

```
/apps
  /web        (existing - DO NOT TOUCH)
  /mobile     (new Expo app)
/services
  /api        (existing backend - reuse only)
/packages
  /shared     (new - shared logic/types if needed)
```

* Place the new Expo app inside `/apps/mobile`
* Ensure complete isolation from `/apps/web`

---

## 2. Mobile App Requirements

Build a **native mobile app using Expo** with the following:

### Core Principles

* The mobile app must use the **existing backend APIs only**
* Do NOT duplicate business logic inside the mobile app
* All validations, workflows, and rules must come from backend responses

---

## 3. API Integration

* Create a centralized API client inside:

  ```
  /apps/mobile/src/api/
  ```
* Use consistent request handling:

  * Authentication (token-based if exists)
  * Error handling
  * Loading states
* Match all request/response structures exactly with backend

---

## 4. Shared Layer (If Needed)

If applicable, create reusable shared resources:

```
/packages/shared/
```

Include:

* Types / Interfaces
* Enums (statuses, roles, etc.)
* Constants
* API contracts

⚠️ Do NOT duplicate logic already handled in backend.

---

## 5. UI/UX Guidelines (Very Important)

* Do NOT copy the web UI directly
* Redesign screens specifically for mobile usage:

  * Simpler navigation
  * Larger touch targets
  * Reduced complexity per screen
* Focus on:

  * Task execution speed
  * Field usability
  * Clear status visibility

---

## 6. Scope of Mobile App

The mobile app must support **the same capabilities as the web app**, but optimized for mobile workflows:

Examples:

* Viewing and updating records
* Changing statuses
* Submitting forms
* Approvals / actions
* Notifications (if applicable)

---

## 7. Navigation

* Use a proper mobile navigation system (e.g., Expo Router or React Navigation)
* Implement structured navigation:

  * Stack
  * Tabs (if needed)
* Keep flows simple and fast

---

## 8. Environment Setup

* Add `.env` support for API base URL
* Do NOT hardcode endpoints

---

## 9. Build & Run

Ensure the app:

* Runs via Expo (`npx expo start`)
* Is ready for EAS build (optional but preferred)

---

## 10. Deliverables

Provide:

1. Folder structure
2. List of implemented screens
3. API endpoints used per screen
4. Any assumptions made
5. Any missing APIs required from backend

---

## Final Rule

The mobile app must be:

* Fully functional
* Fully independent from the web frontend
* Fully dependent on the same backend

No shortcuts, no duplication, no breaking existing system.
